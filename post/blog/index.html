开启我的博客之旅

本文记录了：

  • 我为什么想拥有自己的博客
  • 我为此做了哪些准备
  • 我经历什么。。。
  • 题外话（一点都不题外哦）

一、我为什么想拥有自己的博客

  作为一个刚踏进IT界的渣渣来说，计算机技术带给我的体验不仅仅是让我觉得自己是那么的渺小和卑微，更多是对于这世界的惊叹和向往。小到让我思考了很久，并不断揣摩的快速排序算法，大到诸如git之类的天才般的作品，无论是攻克它们的成就感还是仰视它们的愉悦感，都让我不禁想要做一件事，那就是：记录我和它们的点点滴滴。

  就这样，在2020年的4月18号，我萌生了一个想法：我想拥有自己的博客！

二、我为此做了哪些准备

  我很喜欢耐克那句非常经典的广告语：Just Do It（喜欢到我觉得以后我的每一个博客可能都会以它开始）。

  OK，言归正传，在我决定要创建自己的博客之后。在第一时间，我去百度了一下博客的定义：网络日志…没错，是我想要的东西。(不开玩笑，我真去查了。)然后就是各种搜索怎么创建自己的博客，因为直接在像CSDN这种平台发布博客对我来说是不太能接受的，我总觉得那种博客不是我自己的东西一样。（纯个人问题，不带有任何褒贬！）

   首先，我去了B站，看了几个关于制作博客的视频。大部分视频的标题是“十分钟带你…”、“八分钟让你创建自己的…”这种，这些视频教程在现在看来可能不那么完美，但确实能给一个小白带来一场的洗礼。从这些视频中我获取了以下信息：

  • 当下流行的创建自己的博客的方式有两种：
    • Hexo + Node.js + Git
    • Hugo + Git
  • 我可以把我的博客托管到GitHub Pages
  • 我只需要简单掌握一下MarkDown就可以写自己的博客了
  • 我甚至可以用别人设计好的主题

  原来创建自己的博客那么简单！

  在Hexo和Hugo之间，我选择了Hugo，因为它更轻量，而且足够快捷。

三、我经历了什么。。。

  在决定使用Hugo框架之后，我快速地浏览了Hugo官网，Hugo对自己的定义是：“Hugo is one of the most popular open-source static site generators.“它是一个开源的静态网站生成器，拥有诸多优点，正如它自己介绍的那样，它足够快，足够灵活，足够有趣。然后我很快地完成了Hugo官网上的Quick Start。

  接下来，怀着一颗求知若渴的心，我吭哧瘪肚地看完了Docs里的Getting Started部分和Hugo Modules部分，虽然确实能丰富我的知识，但是似乎对于只想要写个博客的我也没有太大的意义。当然了，对于想要把站点托管在GitHub Pages上的同学来说，Host on GitHub是不容忽略的教程，该教程机智地使用两个远程仓库来分别管理hugo项目和hugo项目生成的public目录，并以“子模块”的方式魔幻般地把作为hugo项目仓库下的一个子文件夹的public目录对接到了GitHub Pages提供的站点仓库，这对于博客的维护和发布来说，简直不要太舒服！这也让我进一步见识到了Git的伟大。此外，该教程提供的使用deploy.sh脚本的方式简化站点更新的操作同样令人感到亲切。

  紧接着，在简单地认识一下MarkDown并知道原来还有Typora这个神器之后，我开始了这篇博客的撰写。

  。。。。。。

四、题外话

  在我创作我的博客网页的过程中，我花费了数个小时重新学习Git的分支，这可能是我在这个过程中最大的收益了。诚然，我使用Git已经有一段时间了，但是由于我几乎没有和别人一起开发过某个项目，导致我基本没有用过git分支的功能，用了一年多git，结果和没用过的效果差不多。当我遇到理解上的问题时，我重新学习了这部分内容，我不得不惊叹：Git的创造者Linus竟然一个人在两周的时间内完成了Git的创造，它是如此简洁，却又如此强大！（我觉得我肯定会安耐不住写一篇关于git分支的博客…）

  遗憾的是，即使我认识到了自己在git使用方面的问题，但我还是不敢奢求能有和别人一起完成某个项目的机会。对此，我只能假装我自己就是那个“合作者”，就像在撰写这篇博客的的时候，我刻意新建了一个mfb分支，然后在这个分支上撰写并测试，最后再把它合并到master上去，尽管这看起来很傻很没必要。
